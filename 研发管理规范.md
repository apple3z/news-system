## 一、核心理念

### 1.1 文档驱动研发
> **"无文档，不开发"**

所有研发活动必须以文档为载体，通过文档进行需求确认、技术方案讨论、测试验证和发布归档。

### 1.2 Web端协同
> **"Web端是唯一的协作界面"**

所有文档的查看、编辑、确认都通过Web端完成，确保信息同步和版本一致。

## 四、文档体系规范

### 4.1 每个版本的12个标准文档

| 序号 | 文档 | 用途 | 必填 | 编写时机 |
|:---:|------|------|:----:|----------|
| 00 | 文档索引.md | 导航和总览 | ✅ | 版本启动时 |
| 01 | 规划.md | 版本目标和计划 | ✅ | 版本启动时 |
| 02 | 需求.md | 功能需求详细描述 | ✅ | 需求分析阶段 |
| 03 | 架构.md | 系统设计 | ✅ | 设计阶段 |
| 04 | 技术.md | 技术实现细节 | ✅ | 设计阶段 |
| 05 | 更新介绍.md | 对外发布说明 | ✅ | 发布前 |
| 06 | 测试计划.md | 测试策略和用例 | ✅ | 测试阶段前 |
| 07 | 测试报告.md | 测试结果 | ✅ | 测试完成后 |
| 08 | 问题清单.md | Bug跟踪 | ✅ | 全程维护 |
| 09 | 迭代日志.md | 开发过程记录 | ✅ | 全程维护 |
| 10 | 发布说明.md | 部署和回滚指南 | ✅ | 发布前 |
| 11 | 状态看板.md | 实时进度跟踪 | ✅ | 全程维护 |

### 4.2 文档命名规范

- **必须使用中文命名**，如 `01-规划.md`
- **必须带序号前缀**，方便排序
- **使用.md后缀**，Markdown格式

### 4.3 文档版本管理

#### 版本号格式
`主版本.次版本.修订号`，如 `1.0.5`

#### 自动版本管理规则
- 每次保存自动递增修订号（第3位）
- 重大变更手动更新次版本（第2位）
- 架构重构手动更新主版本（第1位）

#### 统一版本函数
所有目录下的文档使用统一的版本管理函数：

```python
# 获取文档版本信息
def get_doc_version(filepath):
    """
    支持目录: 版本管理/, 研发规范/
    返回: {'version': '1.0.5', 'mtime': '...', 'history': [...]}
    """
    pass

# 保存文档并更新版本
def save_doc_with_version(filepath, content):
    """
    保存后自动递增版本号
    返回: {'code': 200, 'version': '1.0.6'}
    """
    pass
```

### 5.1 Phase 1：需求协同阶段（Web端协同编辑）

#### 触发条件
张振中提出目标和需求

#### 参与角色
- **主导**: 瑞贝卡（需求分析、文档编写）
- **协同**: 张振中（需求确认、文档修改）

#### 核心流程

**步骤1：需求分析**
- 瑞贝卡分析张振中提出的需求
- 理解业务目标和技术可行性

**步骤2：开启新版本**
```bash
# 在版本管理/下创建新版本目录
mkdir 版本管理/v{X.Y.Z}/
cp 研发规范/模板/* 版本管理/v{X.Y.Z}/
```

**步骤3：编写需求文档**
- 瑞贝卡编写 `02-需求.md`
- 包含：功能描述、用户场景、验收标准

**步骤4：Web端协同确认**
```
瑞贝卡保存文档 ──► Web端自动更新版本 ──► 张振中刷新页面查看最新内容
                                              │
张振中在线编辑修改 ◄── 保存后版本+1 ◄── 直接修改或批注
       │
       ▼
瑞贝卡看到修改 ◄── 刷新页面 ◄── 双方反复协同直至达成一致
```

**关键原则**：
> **"Web端是唯一的协作界面，MD文档是唯一的协作载体"**

**步骤5：需求确认签字**
当双方对需求无歧义后，张振中在文档中签字：
```markdown
## 确认签字
**需求确认**:
- ✅ 已确认 - 张振中 - 2026-03-06
```

#### 阶段产出
- [ ] `02-需求.md` 已完成并确认
- [ ] 需求确认签字已记录
- [ ] 双方对需求理解一致

#### 进入下一阶段条件
✅ **需求文档已确认，无歧义**

### 5.3 Phase 3：体验反馈阶段（循环迭代）

#### 触发条件
研发完成，进入体验环节

#### 参与角色
- **主导**: 张振中（体验、提问题）
- **响应**: 瑞贝卡（收集、修复）

#### 核心流程

**步骤1：全面体验**
张振中进行以下体验：
- 实际使用功能
- 查阅版本管理中的所有文档
- 验证是否符合需求

**步骤2：发现问题**
可能发现的问题类型：

| 问题类型 | 示例 | 处理方式 |
|---------|------|----------|
| **Bug缺陷** | 点击按钮无反应 | 立即修复 |
| **设计缺陷** | 交互流程不合理 | 评估后优化 |
| **需求不完善** | 遗漏了某个场景 | 补充需求 |

**步骤3：提出意见**
- 张振中将问题记录在 `08-问题清单.md`
- 或通过即时通讯紧急沟通

**步骤4：问题处理循环**
```
         ┌─────────────────────────────────────┐
         │                                     │
         ▼                                     │
    ┌─────────┐    ┌─────────┐    ┌─────────┐ │
    │ 收集问题 │───►│ 修改文档 │───►│ 继续研发 │─┘
    └─────────┘    └─────────┘    └─────────┘
                                      │
                                      ▼
                                ┌─────────┐
                                │ 更新测试 │
                                │ 报告等   │
                                └─────────┘
```

**关键原则**：
> **"版本号不变，持续迭代，直至无重大问题"**

**步骤5：循环退出条件**
以下情况退出循环，进入正式发布：
- [ ] 无P0/P1级别Bug
- [ ] 核心功能符合需求
- [ ] 张振中认可质量

#### 阶段产出
- [ ] `08-问题清单.md` 更新（新增问题和修复记录）
- [ ] `07-测试报告.md` 更新（回归测试结果）

#### 进入下一阶段条件

### 5.6 关键成功因素

1. **严格执行文档驱动**
   - 无文档，不开发
   - 文档即代码，同步更新

2. **充分利用Web端协同**
   - 实时查看对方修改
   - 版本历史可追溯

3. **保持版本号稳定**
   - 同一版本内可多次迭代
   - 避免频繁创建新版本

4. **及时更新状态看板**
   - 双方随时了解进度
   - 风险早发现早处理

## 七、协同规范

### 7.1 Web端协同编辑流程

```
1. PM创建/修改文档
        ↓
2. 保存到磁盘（自动触发版本+1）
        ↓
3. 产品负责人刷新Web页面
        ↓
4. 查看最新内容和版本
        ↓
5. 在线编辑批注或直接修改
        ↓
6. 保存（版本再+1）
        ↓
7. 循环直至达成一致
```

### 7.2 沟通方式

| 场景 | 方式 | 响应时间 |
|------|------|----------|
| 需求确认 | Web端协同编辑 | 实时 |
| 问题反馈 | 更新08-问题清单.md | 24小时 |
| 进度同步 | 更新11-状态看板.md | 每日 |
| 紧急事项 | 即时通讯 | 1小时 |

### 7.3 决策机制

| 决策项 | 决策者 | 依据 |
|--------|--------|------|
| 需求变更 | 产品负责人 | 业务价值判断 |
| 技术方案 | PM | 技术可行性评估 |
| 发布时间 | 共同协商 | 质量和进度平衡 |
| 优先级调整 | 产品负责人 | 业务紧急程度 |
