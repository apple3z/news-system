## 四、关键设计决策

### 4.1 为什么采用模块化架构 (v3.0)?
- **可维护性**: 每个模块独立，职责清晰
- **可扩展性**: 新增功能只需添加新的 Blueprint
- **代码组织**: 避免单个 app.py 过于臃肿
- **团队协作**: 不同开发者可并行开发不同模块

### 4.2 为什么不使用数据库存储文档？
- **灵活性**: Markdown 文件便于版本控制 (Git)
- **可读性**: 直接查看和编辑源文件
- **轻量**: 不需要额外的数据库表

### 4.3 为什么实现智能分类 (v2.6)?
- **用户体验**: 用户可以快速找到感兴趣的新闻/Skills
- **信息组织**: 海量内容需要有效分类
- **未来扩展**: 为推荐系统打基础

### 4.4 为什么用外部 JS 文件？
- **问题**: Flask 渲染的 HTML 中内联 `<script>` 有时不执行
- **解决**: 将 JS 放到 `/static/sys_v2.js`用`<script src>`引用
- **优势**: 浏览器缓存，减少页面加载时间

## 五、安全考虑

### 5.1 文件操作
- 路径限制：只能在 `doc/` 目录下操作
- 文件类型：只允许 `.md` 文件
- 特殊字符：过滤 `..` 防止路径遍历

### 5.2 锁定机制
- 只增不减：锁定后不能减少内容
- 提示用户：明确告知锁定状态

### 5.3 API 安全
- CSRF 保护：Flask secret_key
- 输入验证：所有 API 参数都经过验证
- 错误处理：统一返回格式，不暴露内部错误

## 六、性能优化

### 6.1 页面加载
- 静态资源：CSS/JS 使用本地静态目录
- 图片：使用封面图 URL 而非 Base64
- 缓存：浏览器缓存静态资源

### 6.2 数据库
- 索引：已对 id、category、created_at 字段建立索引
- 缓存：内存缓存热点数据 (Python dict)
- 分页：所有列表接口都支持分页

### 6.3 搜索优化 (v2.6)
- LIKE 查询：使用索引优化
- 分页：限制每次查询数量
- 关键词：支持多字段组合搜索

### 7.2 新增 Skills 分类
```python
CATEGORY_KEYWORDS = {
    '新分类': ['关键词 1', '关键词 2', ...],
    # ...
}

def classify_skill(text):
    scores = {}
    for category, keywords in CATEGORY_KEYWORDS.items():
        scores[category] = sum(1 for kw in keywords if kw in text)
    return max(scores, key=scores.get)
```
