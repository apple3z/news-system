## 一、核心理念

### 1.1 文档驱动研发
> **"无文档，不开发"**

所有研发活动必须以文档为载体，通过文档进行需求确认、技术方案讨论、测试验证和发布归档。

### 1.2 Web 端协同
> **"Web 端是唯一的协作界面"**

所有文档的查看、编辑、确认都通过 Web 端完成，确保信息同步和版本一致。

### 4.2 文档命名规范

- **必须使用中文命名**，如 `01-规划.md`
- **必须带序号前缀**，方便排序
- **使用.md 后缀**，Markdown 格式

### 4.3 文档版本管理

#### 版本号格式
`主版本。次版本。修订号`，如 `1.0.5`

#### 自动版本管理规则
- 每次保存自动递增修订号（第 3 位）
- 重大变更手动更新次版本（第 2 位）
- 架构重构手动更新主版本（第 1 位）

## 五、研发流程规范（核心）

### 5.0 核心理念：三大阶段，一个循环

```
Phase 1: 需求协同 (Web 端 MD 文档协同编辑 → 签字确认)
    ↓
Phase 2: 研发实施 (按规范执行 → 全程文档更新)
    ↓
Phase 3: 体验反馈 (张总体验 → 问题循环迭代)
    ↓
Release: 正式发布 (版本归档 → 部署上线)
```

### 5.1 Phase 1：需求协同阶段

**触发条件**: 张总提出目标和需求

**参与角色**:
- **主导**: 瑞贝卡（需求分析、文档编写）
- **协同**: 张总（需求确认、文档修改）

**核心流程**:
1. 瑞贝卡分析需求
2. 开启新版本（在 `版本历史/` 下创建版本目录）
3. 编写 `当前版本的需求.md`
4. **Web 端协同确认**（张总通过 Web 端查看、修改、批注）
5. 需求通过飞书或者webchat确认

**关键原则**:
> **"Web 端是唯一的协作界面，MD 文档是唯一的协作载体"**

### 5.2 Phase 2：研发实施阶段

**触发条件**: 需求已确认

**参与角色**:
- **主导**: 瑞贝卡（全流程执行）
- **执行**: 小 Q（编码、测试）
- **监督**: 张总（进度查看）

**核心流程**:
1. 阅读《研发管理规范.md》
2. 按规范执行开发（架构→技术→编码→测试）
3. 全程更新文档（迭代日志、状态看板、问题清单）
4. 完成发布

### 5.3 Phase 3：体验反馈阶段

**触发条件**: 研发完成，进入体验环节

**参与角色**:
- **主导**: 张总（体验、提问题）
- **响应**: 瑞贝卡（收集、修复）

**核心流程**:
1. 张总全面体验（查阅文档 + 实际使用）
2. 发现问题（Bug、设计缺陷、需求不完善）
3. 记录问题到 `08-问题清单.md`
4. 瑞贝卡修复问题，更新文档
5. 循环迭代直至无重大问题

### 5.4 Release：正式发布

**触发条件**: 版本研发完毕，质量达标

**执行动作**:
1. 获取 Release 版本
2. 版本正式发布
3. 本次研发结束

## 七、协同规范

### 7.1 Web 端协同编辑流程

```
1. PM 创建/修改文档
        ↓
2. 保存到磁盘（自动触发版本 +1）
        ↓
3. 产品负责人刷新 Web 页面
        ↓
4. 查看最新内容和版本
        ↓
5. 在线编辑批注或直接修改
        ↓
6. 保存（版本再 +1）
        ↓
7. 循环直至达成一致
```

| 场景 | 方式 | 响应时间 |
|------|------|----------|
| 需求确认 | Web 端协同编辑 | 实时 |

| 进度同步 | 更新 11-状态看板.md | 每日 |
| 紧急事项 | 即时通讯 | 1 小时 |

### 7.3 决策机制

| 决策项 | 决策者 | 依据 |
|--------|--------|------|
| 需求变更 | 产品负责人 | 业务价值判断 |

| 发布时间 | 共同协商 | 质量和进度平衡 |

## 八、PM 管理规范（2026-03-09 新增）

### 8.1 5 分钟进度检查规范

**核心理念**: PM 必须主动驱动项目，不要等待小 Q 确认。

**规范内容**:
1. **每隔 5 分钟**，通过 CLI 了解小 Q 的研发进度
2. **主动驱动**项目进度继续
3. **需要解决的问题和选择**，PM 来做决定
4. **不要等待**小 Q 确认，主动检查并确认

**执行方式**:
```bash
# 检查小 Q 会话进度
process(action=log, sessionId=<session_id>)

# 根据进度决定下一步
# 需要修复 → 创建新任务
# 已完成 → 确认并更新文档
# 阻塞 → 解决问题
```

**PM 职责**:
- 主动检查进度（每 5 分钟）
- 及时确认小 Q 的工作成果
- 解决阻塞问题
- 更新项目文档（迭代日志、状态看板、测试报告）
- 驱动项目持续前进

**文档更新**:
- 每次检查后更新 `09-迭代日志.md`
- 每次里程碑更新 `11-状态看板.md`
- Bug 修复后更新 `07-测试报告.md` 和 `08-问题清单.md`
