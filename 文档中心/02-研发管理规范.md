## 一、核心理念

### 1.1 文档驱动研发
> **"无文档，不开发"**

所有研发活动必须以文档为载体，通过文档进行需求确认、技术方案讨论、测试验证和发布归档。

### 1.2 Web端协同
> **"Web端是唯一的协作界面"**

所有文档的查看、编辑、确认都通过Web端完成，确保信息同步和版本一致。

### 1.3 前端/后端分离开发模式
> **"明确分工，协作有序"**

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         开发流程总览                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   【阶段1】产品经理设计需求                                              │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  1.1 设计功能需求                                              │   │
│   │  1.2 编写需求文档 (02-需求.md)                                │   │
│   │  1.3 明确用户体验和交互流程                                    │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                   ▼                                    │
│   【阶段2】前端开发 + API需求提出                                       │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  2.1 根据需求开发前端页面和组件                                │   │
│   │  2.2 定义前端需要的 API 接口                                   │   │
│   │  2.3 编写 API 需求文档 (明确请求/响应格式)                     │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                   ▼                                    │
│   【阶段3】API 需求确认（必须在此阶段暂停后端开发）                      │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  3.1 产品经理确认前端需求                                       │   │
│   │  3.2 前端提交 API 需求文档                                     │   │
│   │  3.3 产品经理确认 API 需求                                     │   │
│   │  ⚠️ 只有确认后才能开启后端开发                                  │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                   ▼                                    │
│   【阶段4】后端API开发（不再参与前端开发）                              │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  4.1 根据已确认的 API 需求实现后端接口                          │   │
│   │  4.2 完成业务逻辑和数据访问层                                   │   │
│   │  4.3 编写单元测试                                              │   │
│   │  ⚠️ 后端完成前不参与前端开发                                    │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                   ▼                                    │
│   【阶段5】后端联调（后端统一完成）                                     │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  5.1 后端完成所有 API 实现后                                   │   │
│   │  5.2 后端进行前后端联调                                        │   │
│   │  5.3 修复联调过程中发现的问题                                  │   │
│   │  5.4 完成后端不再参与前端修改                                  │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**关键规则**：
1. **前端优先**：前端先完成开发，提出明确的API需求
2. **API确认制**：产品经理确认API需求后，后端才能开始开发
3. **后端独立**：后端开发期间不再参与前端修改
4. **联调统一**：联调由后端统一完成，前端配合验证
5. **文档同步**：每次迭代后，文档中心必须同步更新

| 序号 | 文档 | 用途 | 必填 | 编写时机 |

### 4.2 文档命名规范

- **必须使用中文命名**，如 `01-规划.md`
- **必须带序号前缀**，方便排序
- **使用.md后缀**，Markdown格式

### 4.3 文档版本管理

#### 版本号格式
`主版本.次版本.修订号`，如 `1.0.5`

#### 自动版本管理规则
- 每次保存自动递增修订号（第3位）
- 重大变更手动更新次版本（第2位）
- 架构重构手动更新主版本（第1位）

#### 统一版本函数
所有目录下的文档使用统一的版本管理函数：

# 保存文档并更新版本

    """

### 5.0 核心理念：三大阶段，一个循环

### 5.1 Phase 1：需求协同阶段（Web端协同编辑）

#### 触发条件
张振中提出目标和需求

#### 参与角色
- **主导**: 瑞贝卡（需求分析、文档编写）
- **协同**: 张振中（需求确认、文档修改）

#### 核心流程

**步骤1：需求分析**
- 瑞贝卡分析张振中提出的需求
- 理解业务目标和技术可行性

**步骤2：开启新版本**
```bash
# 在版本管理/下创建新版本目录
mkdir 版本管理/v{X.Y.Z}/
cp 研发规范/模板/* 版本管理/v{X.Y.Z}/
```

**步骤3：编写需求文档**
- 瑞贝卡编写 `02-需求.md`
- 包含：功能描述、用户场景、验收标准

**步骤4：Web端协同确认**
```
瑞贝卡保存文档 ──► Web端自动更新版本 ──► 张振中刷新页面查看最新内容
                                              │
张振中在线编辑修改 ◄── 保存后版本+1 ◄── 直接修改或批注
       │
       ▼
瑞贝卡看到修改 ◄── 刷新页面 ◄── 双方反复协同直至达成一致
```

**关键原则**：
> **"Web端是唯一的协作界面，MD文档是唯一的协作载体"**

**步骤5：需求确认签字**
当双方对需求无歧义后，张振中在文档中签字：
```markdown
**需求确认**:
- ✅ 已确认 - 张振中 - 2026-03-06
```

#### 阶段产出
- [ ] `02-需求.md` 已完成并确认
- [ ] 需求确认签字已记录
- [ ] 双方对需求理解一致

#### 进入下一阶段条件
✅ **需求文档已确认，无歧义**

### 5.3 Phase 3：体验反馈阶段（循环迭代）

#### 触发条件
研发完成，进入体验环节

#### 参与角色
- **主导**: 张振中（体验、提问题）
- **响应**: 瑞贝卡（收集、修复）

#### 核心流程

**步骤1：全面体验**
张振中进行以下体验：
- 实际使用功能
- 查阅版本管理中的所有文档
- 验证是否符合需求

**步骤2：发现问题**
可能发现的问题类型：

| 问题类型 | 示例 | 处理方式 |
|---------|------|----------|
| **Bug缺陷** | 点击按钮无反应 | 立即修复 |
| **设计缺陷** | 交互流程不合理 | 评估后优化 |
| **需求不完善** | 遗漏了某个场景 | 补充需求 |

**步骤3：提出意见**
- 张振中将问题记录在 `08-问题清单.md`
- 或通过即时通讯紧急沟通

**步骤4：问题处理循环**
```
         ┌─────────────────────────────────────┐
         │                                     │
         ▼                                     │
    ┌─────────┐    ┌─────────┐    ┌─────────┐ │
    │ 收集问题 │───►│ 修改文档 │───►│ 继续研发 │─┘
    └─────────┘    └─────────┘    └─────────┘
                                      │
                                      ▼
                                ┌─────────┐
                                │ 更新测试 │
                                │ 报告等   │
                                └─────────┘
```

**关键原则**：
> **"版本号不变，持续迭代，直至无重大问题"**

**步骤5：循环退出条件**
以下情况退出循环，进入正式发布：
- [ ] 无P0/P1级别Bug
- [ ] 核心功能符合需求
- [ ] 张振中认可质量

#### 阶段产出
- [ ] `08-问题清单.md` 更新（新增问题和修复记录）
- [ ] `07-测试报告.md` 更新（回归测试结果）
- [ ] `02-需求.md` 可能更新（需求补充）

#### 进入下一阶段条件
✅ **无重大问题，版本研发完毕**

## 八、质量标准

### 8.1 文档质量

- [ ] 结构清晰，层次分明
- [ ] 内容完整，无遗漏
- [ ] 格式统一，遵循模板
- [ ] 及时更新，与实际一致

### 8.2 代码质量

- [ ] 遵循编码规范
- [ ] 单元测试覆盖率>80%
- [ ] 无P0/P1级别Bug
- [ ] 代码审查通过

#### 8.2.1 代码审查规范

| 审查项 | 要求 | 责任人 |
|--------|------|--------|
| 代码规范 | PEP8/ESLint检查通过 | 开发者 |
| 安全性 | 无敏感信息泄露、无SQL注入 | 开发者 |
| 性能 | 无N+1查询、无内存泄漏 | 开发者 |
| 可读性 | 命名规范、注释清晰 | 审查者 |
| 测试 | 单元测试>80%覆盖 | 开发者 |

**审查流程**：
```
1. 开发者提交代码 → 2. 自动流水线检查 → 3. 代码审查 → 4. 反馈修改 → 5. 通过合并
```

### 8.3 发布质量

- [ ] 所有P0需求已实现
- [ ] 所有测试用例通过
- [ ] 性能指标达标
- [ ] 产品负责人验收签字

### 8.4 测试规范

#### 8.4.1 测试分层

| 层级 | 类型 | 覆盖率要求 | 执行时机 |
|------|------|-----------|----------|
| 单元测试 | 函数/方法级 | >80% | 每次提交 |
| 集成测试 | 模块级接口 | >60% | 每日构建 |
| 端到端测试 | 业务流程 | 核心场景 | 发布前 |

#### 8.4.2 测试通过标准

| 级别 | 通过标准 | 阻塞发布 |
|------|----------|----------|
| P0 | 必须通过 | ✅ 阻塞 |
| P1 | 必须通过 | ✅ 阻塞 |
| P2 | 建议通过 | ⚠️ 评估 |
| P3 | 可选 | ❌ 不阻塞 |

#### 8.4.3 Bug等级定义

| 等级 | 定义 | 示例 |
|------|------|------|
| P0 | 系统崩溃、数据丢失 | 服务无法启动、数据错误 |
| P1 | 核心功能不可用 | 登录失败、保存失败 |
| P2 | 功能异常但有替代方案 | 页面显示错位 |
| P3 | 体验问题 | 文字拼写错误 |
