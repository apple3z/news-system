## 一、文档驱动的研发流程

### 1.1 文档存放
- **物理地址**: `/home/zhang/.copaw/news_system/docs/
- **访问地址**: http://192.168.36.132:5000/sys

### 1.2 角色分工

| 角色 | 职责 |
|------|------|
| **AI研发人员** | 阅读文档、撰写PRD/ARCH/TDD等技术文档，执行研发实现 |
| **用户（你）** | 查看项目文档、确认需求、审核研发成果 |

### 1.3 研发流程

```
1. AI阅读规范文档 → 了解项目规范和历史版本
       ↓
2. 需求确认 → 用户提出需求
       ↓
3. 撰写文档 → AI按照规范撰写全部PRD/ARCH/TDD等文档
       ↓
4. 用户审核 → 通过 http://192.168.36.132:5000/sys 查看全部文档
       ↓
5. 确认无误 → 用户确认后进入研发实现阶段
       ↓
6. 研发实现 → AI按照文档进行技术开发
       ↓
7. 测试验证 → 测试用例、测试报告
       ↓
8. 发布运营 → 源代码归档、运维报告
```

### 1.4 审核机制

- **文档审核**: 用户通过系统管理查看所有文档，确认需求
- **研发审核**: 每个阶段完成后需要用户确认才能进入下一阶段
- **发布审核**: 源代码和测试报告确认后才能发布

### 1.5 文档操作

- **AI操作**: 直接读写 `/home/zhang/.copaw/news_system/docs/` 目录下的 .md 文件
- **用户查看**: 通过 http://192.168.36.132:5000/sys 系统管理页面查看
- **文档编辑**: 在系统管理页面点击编辑按钮进行在线编辑
- **文档锁定**: 被标记为锁定的文档，内容只能添加不能删减

## 四、文档撰写规范

### 4.1 文档命名规则
```
{文档类型}_{版本号}.md

示例：
PRD_v2.4.md      - 需求文档
ARCH_v2.4.md     - 架构设计
TDD_v2.4.md      - 技术设计
TEST_PLAN_v2.4.md - 测试计划

```

```
## 五、数据模型
```

```

## 三、数据架构

```

```

## 二、接口设计
## 三、核心实现
## 四，安全设计
```

## 六、文档管理

### 6.1 文档存储位置
```
news_system/docs/
├── 研发管理规范.md        # 本文档 - 总体研发规范（锁定）
├── 文档撰写规范.md        # 文档撰写标准
├── PRD_v{X}.md           # 需求文档（按版本）
├── ARCH_v{X}.md          # 架构文档（按版本）
├── TDD_v{X}.md           # 技术文档（按版本）
├── CODING_v{X}.md        # 代码设计（按版本）
├── DB_v{X}.md            # 数据库设计（按版本）
├── TEST_PLAN_v{X}.md     # 测试计划（按版本）
├── TEST_CASE_v{X}.md     # 测试用例（按版本）
└── DEV_PLAN_v{X}.md      # 研发计划（按版本）
```

### 6.2 文档锁定功能
- 被标记为**锁定**的文档，内容只能添加不能删减
- 锁定文档在编辑器会有提示
- 锁定标记保存在版本记录文件中

### 6.3 文档版本追溯
- 每个版本独立维护文档
- 文档命名带版本号
- 更新记录在更新介绍中

## 九、版本号自动管理

### 9.1 版本号格式
- **格式**: XX.XX.XX（语义化版本）
- **示例**: 1.0.0, 1.0.1, 1.1.0, 2.0.0

### 9.2 自动生成规则
- 每次文档保存时，版本号自动递增修订号（第三位）
- 版本号和时间由系统自动管理，**无需在文档内容中手动标记**
- 系统自动提取文档内容中的版本标识进行匹配更新

### 9.3 版本信息显示
- 系统自动从文档中提取版本号和时间显示在界面顶部
- 用户可通过界面查看版本信息，无需手动维护
