## 问题详情

### BUG-002: 后台新建数据源失败

**根因**: `datasource_dal.py` 的 `create_source()` 方法无 try/except，数据库错误直接返回 HTML 500

**修复**: 给 `create_source()`、`update_source()`、`delete_source()` 加了 try/except 错误处理，返回 JSON 格式的错误信息

---

### BUG-003: 新闻卡片时间全显示"1分钟"

**根因（两个）**:
1. `news_dal.py` 的 `search_news()` SELECT 不包含 `content` 字段，前端 `getReadingTime(item.content)` 收到 undefined → 始终返回 "1分钟"
2. 卡片模板从未渲染 `item.time`（发布日期），只显示阅读时间

**修复**:
- `news_dal.py` — SELECT 加入 `content` 和 `image` 字段
- `NewsView.js` — 卡片 meta 区域新增显示 `item.time`（发布日期）

---

### BUG-004: 热门排行出现2023年旧新闻

**根因**: 热门查询 `sort_by=hot_score` 没有时间过滤，hot_score 无时间衰减

**修复**: `NewsView.js` — `loadHotNews()` 加了 `date_from` 参数（近30天）

---

### BUG-005: 只有163和新浪有数据

**根因（三个叠加）**:
1. 36kr/虎嗅/机器之心/量子位/CSDN等站点是JS渲染SPA，`requests.get()` + BeautifulSoup 拿不到内容
2. `get_page()` 用 bare `except` 吞掉所有错误，静默失败
3. `unique[:30]` 硬限制，163和新浪先调用占满配额

**修复**:
- 7个JS渲染站改为 RSS Feed 采集：36氪、虎嗅、机器之心、新智元、量子位、InfoQ、CSDN
- 新增 `fetch_from_rss()` 通用RSS解析函数（用 `xml.etree.ElementTree`，无新依赖）
- 新增 `parse_rss_time()` 解析 ISO 8601 + RFC 822 时间格式
- `get_page()` 改为打印错误日志，不再静默失败
- 配额从 `unique[:30]` 提升到 `unique[:100]`
- RSS源优先使用自带的时间和摘要数据

---

- `fetch_news()` 每条新闻记录 `{title, link, source, time, summary}` 存入 `crawl_log.error_message` 字段（JSON）

- `NewsAdmin.js` 日志表格新增"详情"列，可展开查看每条新闻的基础信息

### BUG-014: Skills 数据重复入库

**根因**: `fetch_skills.py` 用 `name + owner` 做 UPSERT 查重，但旧数据（id 1-10）和新数据（id 101-110）的 owner 在某些情况下不一致，导致同一个 skill 被插入两次

**修复**:
- 在 `fetch_skills()` 入口添加 `deduplicate_skills()` 函数，按 `name + owner` 分组只保留最新记录
- UPSERT 逻辑保持 `name + owner` 查重，确保后续采集不再重复

---

### BUG-015: stars 全部为 0

**根因**: GitHub API 未认证每小时限 60 次，爬虫连续请求被限流；旧数据初始导入时未调用 GitHub API

**修复**:
- `fetch_github_meta()` 增加重试机制和 403 限流检测
- 增加每次请求间 1 秒延迟避免触发限流
- 手动补充 10 个仓库的真实 stars 数据

---

### BUG-016: readme_content 全为 "404: Not Found"

**根因**: 旧代码用 `{gh_owner}/{gh_repo}/main/README.md` 固定路径，但部分仓库默认分支是 `master`；且 `fetch_url()` 对 404 响应未正确过滤

**修复**:
- 新增 `fetch_readme()` 函数，从 GitHub API 获取 `default_branch`，按正确分支名拉取 README
- 增加 fallback：先试 default_branch，再试 master/main
- `fetch_url()` 只在 HTTP 200 时返回内容

---

### BUG-017: 详情页大量字段为 None

**根因**: 爬虫 INSERT/UPDATE 语句未包含 tags, languages, tech_stack 等字段

**修复**:
- `fetch_skills.py` 的 INSERT 和 UPDATE 语句增加 `tags, languages, tech_stack` 字段
- 从 GitHub API 的 `topics`、`language` 字段提取数据填充

---

### BUG-018: list_skills (admin) 缺少 stars/downloads

**根因**: `skills_dal.py` 的 `list_skills()` SELECT 语句遗漏 stars 和 downloads

**修复**: SELECT 增加 `stars, downloads` 字段

---

### BUG-019: CRUD 无错误处理

**根因**: `create_skill()`、`update_skill()`、`delete_skill()` 均无 try/except

**修复**: 三个方法均增加 try/except/finally，异常时 rollback 并抛出，finally 中关闭连接

---

### BUG-020: search_skills_public 返回字段不足

**根因**: SELECT 只返回 8 个字段，缺少 url, github_url, skill_level, features

**修复**: SELECT 增加 `url, github_url, skill_level, features`，共返回 12 个字段；改用 `Row` 工厂返回 dict

---

### BUG-021: chinese_intro 内容模板化

**根因**: 旧数据用固定模板 `「xxx」是专业型工具，擅长网页等` 生成

**修复**: 新增 `build_chinese_intro()` 函数，整合 name + GitHub description + stars + level + features 生成有意义的描述

---

### BUG-022: Skills 工具市场 Tab 切换无效

**根因（两个）**:
1. `skills_dal.py` 的 `search_skills_public()` 返回字段缺少 `created_at`，"最新发布"Tab 排序时所有 `created_at` 都是 `undefined`，排序无效果
2. "热门推荐"Tab 过滤条件 `stars > 50` 门槛太低，50 个 Skills 几乎全部满足，和"全部工具"看起来一样

**修复**:
- `skills_dal.py` — `search_skills_public()` 的 SELECT 增加 `created_at` 字段
- `SkillsView.js` — 热门推荐过滤条件改为 `stars >= 10000`（约 30 个满足），并按 stars 降序排序

## 问题详情（新增）

### BUG-023: RSS源 sub_type 设置错误

**根因**: 后台添加RSS订阅时，sub_type 被错误设置为 'website' 而不是 'rss'

**影响**: 
- 不会走 feedparser 解析分支
- 回退到纯文本抓取逻辑
- 导致所有结构化数据丢失

**修复方案**: 
1. 批量更新数据库中错误的 sub_type
2. 修复后台管理添加订阅时的类型选择

---

### BUG-024 ~ BUG-027: RSS数据保存错误

**根因**: `subscribe_manager.py` 中的数据保存逻辑有缺陷

**具体问题**:
1. **BUG-024**: `content` 字段存的是 `summary`，应该优先存完整内容
2. **BUG-025**: `summary` 被强制截断到500字符，但实际RSS返回完整HTML
3. **BUG-026**: `title` 字段在某些情况下存的是订阅源名称
4. **BUG-027**: `link` 字段存的是RSS源地址，不是文章链接

**修复方案**: 详见 `17-RSS数据采集修复报告.md`

---

### REQ-012 ~ REQ-016: API需求

**详细需求**: 详见 `15-订阅功能API需求.md`

**API清单**:

4. **REQ-015**: `GET /api/subscribe/subscriptions` - 获取订阅源详情
5. **REQ-016**: RSS采集增强 - 提取 author 和 thumbnail
