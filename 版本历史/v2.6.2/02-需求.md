## 一、现状问题分析

### 1.1 爬虫体系问题

| 问题 | 现状 | 期望 |
|------|------|------|
| 爬虫割裂 | 新闻爬虫（后台线程）、Skills爬虫（子进程）、RSS检查（后台线程）各自独立 | 统一为"数据采集工作台"，一键或分类采集 |
| 源管理缺失 | 新闻源硬编码在 fetch_news.py（10个函数），Skills源硬编码在 fetch_skills.py（10个URL） | 数据源可动态管理（增删改查），存储在数据库 |
| 采集不全面 | "启动爬虫"只爬新闻 | "启动采集"应覆盖全部数据类型 |

### 1.2 前台功能问题

| 问题 | 现状 | 期望 |
|------|------|------|
| 订阅页定位错误 | SubscribeView 展示订阅列表+删除按钮，实质是管理功能 | 应展示 RSS 订阅的实际内容（文章列表） |

| 新闻页功能基本完善 | 有搜索/筛选/排序/轮播/排行 | 小幅优化即可 |

### 1.3 后台管理问题

| 问题 | 现状 | 期望 |
|------|------|------|
| 列表不统一 | NewsAdmin只有爬虫日志表；SkillsAdmin有完整CRUD表+搜索；SubscribeAdmin有CRUD+状态切换+历史 | 三个管理列表统一风格和操作 |
| 无账户管理 | 任何人可访问 /sys | 登录认证 + 账户管理 |

## 二、功能需求设计

### 2.1 模块1：数据采集工作台（后台 /sys/news 重构）

#### 2.1.1 概述
将现有"新闻管理"重构为"数据采集工作台"，整合新闻、Skills、RSS订阅的采集功能和数据源管理。

#### 2.1.2 工作台布局

```
┌─────────────────────────────────────────────────────────────┐
│  数据采集工作台                                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐      │
│  │ 新闻源   │  │ Skills源 │  │ RSS源    │  │ 采集日志 │      │
│  │  12个    │  │  10个    │  │  5个     │  │ 最近10条 │      │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘      │
│                                                             │
│  ┌──────────────────────────────────────────────────┐      │
│  │ [▶ 全量采集]  [📰 采集新闻] [🔧 采集Skills] [📡 采集RSS] │      │
│  └──────────────────────────────────────────────────┘      │
│                                                             │
│  Tab: [数据源管理] [采集日志]                                │
│                                                             │
│  ┌──────────────────────────────────────────────────┐      │
│  │  数据源管理                                       │      │
│  │  筛选: [全部▼] [新闻源/Skills源/RSS源]  [+ 添加源] │      │
│  │                                                   │      │
│  │  ID | 名称 | 类型 | URL | 状态 | 上次采集 | 操作  │      │
│  │  1  | 网易  | 新闻 | ... | 启用 | 03-12    | 编辑/删除│    │
│  │  2  | 36氪  | 新闻 | ... | 启用 | 03-12    | 编辑/删除│    │
│  │  ...                                              │      │
│  └──────────────────────────────────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 2.1.3 功能点

**F1.1 统计卡片**
- 新闻源数量
- Skills源数量
- RSS源数量
- 最近采集日志条数

**F1.2 采集操作按钮**
- **全量采集**：依次采集新闻 → Skills → RSS，后台线程执行
- **采集新闻**：仅采集新闻源
- **采集Skills**：仅采集Skills源
- **采集RSS**：仅检查RSS订阅更新

**F1.3 数据源管理（Tab 1）**

统一管理所有数据源（新闻源、Skills源、RSS源），存储在数据库中。

| 字段 | 说明 |
|------|------|
| id | 自增主键 |
| name | 数据源名称 |
| type | 类型：news / skill / rss |
| url | 数据源URL |
| config | JSON配置（抓取规则、关键词过滤等） |
| status | 状态：active / inactive |
| priority | 优先级（1-20） |
| last_crawl_time | 上次采集时间 |
| created_at | 创建时间 |

操作：
- 新增数据源（弹窗表单）
- 编辑数据源
- 删除数据源
- 启用/禁用切换
- 按类型筛选

**F1.4 采集日志（Tab 2）**

统一展示所有类型的采集日志。

| 字段 | 说明 |
|------|------|
| id | 自增主键 |
| type | 采集类型：news / skill / rss / full |
| start_time | 开始时间 |
| end_time | 结束时间 |
| status | 状态：running / completed / failed |
| total_count | 总数 |
| new_count | 新增数 |
| updated_count | 更新数 |
| detail | 详细信息（JSON） |

### 2.4 模块4：账户管理

#### 2.4.1 数据库设计

```sql
CREATE TABLE admin_users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,    -- SHA256 + salt
    salt TEXT NOT NULL,
    display_name TEXT,
    role TEXT DEFAULT 'admin',      -- admin / super_admin
    status TEXT DEFAULT 'active',   -- active / inactive
    last_login TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP
);
```

#### 2.4.2 功能点

**F4.1 登录页面**
- 路由：/sys/login
- 用户名 + 密码
- 登录后通过 Flask Session 维持会话
- 登录失败提示

**F4.2 登录状态**
- 后台页面顶部显示当前用户
- 登出按钮

**F4.3 账户管理（超级管理员）**
- 路由：/sys/accounts（新Tab，放在"基础管理"下）
- 账户列表：用户名、显示名、角色、状态、上次登录、操作
- 新增账户
- 编辑账户（修改显示名、角色、重置密码）
- 禁用/启用账户
- 删除账户

**F4.4 后台路由鉴权**
- Flask 中间件：所有 /api/admin/* 接口需要登录
- 未登录访问 /sys 自动跳转到 /sys/login
- Session 超时：30分钟无操作自动登出

**F4.5 初始账户**
- 系统初始化时创建默认超级管理员：
  - 用户名：admin
  - 密码：admin123（首次登录强制修改）

## 三、非功能需求

### 3.1 兼容性
- 保持现有公开API不变
- 爬虫重构需兼容现有数据（news.db, skills.db, subscribe.db）

### 3.2 性能
- 全量采集需支持后台异步执行，不阻塞Web请求
- 采集过程实时更新状态

### 3.3 安全
- 密码 SHA256 + 盐加密
- Session HttpOnly
- 防暴力破解（登录失败5次锁定15分钟）
