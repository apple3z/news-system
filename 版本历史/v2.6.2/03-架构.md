### 2.4 API 封装层 (api.js)

```
API 对象
├── get(url)          → GET 请求
├── post(url, data)   → POST 请求
├── put(url, data)    → PUT 请求
├── del(url)          → DELETE 请求
└── _request(url, options)
    ├── 成功 → 返回 JSON
    ├── 401 (非auth路径) → 自动跳转 /sys/login
    └── 网络错误 → { code: 500, message: '...' }
```

### 2.5 认证状态管理

- 登录成功 → 存入 `window.__currentUser`
- AdminLayout 挂载时 → 调用 `/api/auth/me` 验证
- 验证失败/401 → 重定向 `/sys/login`
- 页面刷新 → 依赖 Flask Session Cookie 恢复

### 3.2 三层架构调用链

```
HTTP Request
    │
    ▼
Routes (路由层)          → 解析请求参数, 调用 Service, 封装 JSON 响应
    │                      装饰器: @login_required, @super_admin_required
    ▼
Services (业务逻辑层)    → 纯函数, 无 HTTP 依赖, 可被 proxy 跨模块调用
    │
    ▼
DAL (数据访问层)         → SQL 操作, 返回 dict/list, 负责连接管理
    │
    ▼
SQLite 数据库 / 文件系统
```

### 3.3 认证系统 (v2.6.2 新增)

```python
# utils/auth.py
@login_required          # 检查 session['user'], 401 则返回未认证
@super_admin_required    # 额外检查 role == 'super_admin'
```

保护范围:
- `/api/admin/*` 所有管理端点 → `@login_required`
- `/api/admin/users/*` 用户管理 → `@super_admin_required`
- `/api/auth/login` / `/api/auth/me` → 无需认证
- `/api/news/*` / `/api/skills/*` 公开端点 → 无需认证

## 4. 数据采集架构 (v2.6.2 重点)

### 4.1 采集引擎设计

```
crawler_service.start_unified_crawl(crawl_type)
    │
    ├── type='full'  → 同时执行 news + skill
    ├── type='news'  → fetch_news()
    ├── type='skill' → fetch_skills()
    └── type='rss'   → fetch_news() (只采集RSS源)
    │
    ▼
fetch_news.py
    ├── 硬编码源 (HTML门户):
    │   ├── fetch_163()     → 网易科技 (requests + BS4)
    │   └── fetch_sina()    → 新浪科技 (requests + BS4)
    ├── 硬编码源 (RSS):
    │   ├── fetch_36kr()    → 36氪
    │   ├── fetch_huxiu()   → 虎嗅
    │   ├── fetch_jqr()     → 机器之心
    │   ├── fetch_xz2()     → 新智元
    │   ├── fetch_qbit()    → 量子位
    │   ├── fetch_infoq()   → InfoQ
    │   └── fetch_csdn()    → CSDN
    ├── 动态源 (system.db):
    │   └── fetch_dynamic_rss_sources()
    │       → 从 data_sources 表读取 type='rss' 的源
    │       → 自动跳过硬编码源名称
    │       → 调用 fetch_from_rss() 通用解析
    └── 通用 RSS 解析器:
        └── fetch_from_rss(url, source_name, priority)
            → 支持 RSS 2.0 + Atom 格式
            → parse_rss_time() 时间解析
```

### 4.2 数据源分类

| type | 含义 | 采集方式 | 示例 |
|------|------|----------|------|
| `news` | 新闻门户 | requests + BS4 HTML解析 | 网易科技、新浪科技 |
| `rss` | RSS订阅 | XML RSS/Atom 解析 | 36氪、TechCrunch、Hacker News |
| `skill` | Skill门户 | 专用爬虫 | GitHub Trending 等 |

### 4.3 采集流程

```
1. 用户点击"开始采集" → POST /api/admin/crawl/start {type}
2. crawler_service 调度 → 调用对应采集函数
3. 采集函数返回新闻列表 → [{title, link, source, category, time, summary}]
4. 去重 (按 title[:30] 唯一)
5. 均衡选取 → 每源保底10条 + 优先级补满至200条上限
6. 逐条获取详情 → 更新已有 / 插入新增
5. 记录日志到 crawl_log 表 → {total, new, updated, detail}
6. 返回采集结果 → 前端刷新日志列表
```

## 9. 架构问题清单

> 以下为架构审查中发现的设计不合理之处和已知 BUG，详见 [08-问题清单.md](./08-问题清单.md)

### 9.1 BUG 类

| # | 严重度 | 问题 | 位置 |
|---|--------|------|------|
| B1 | 🔴 高 | SkillDetailView 公开页面调用 `/api/admin/skills/:id` (管理端点)，未登录用户会被 401 跳转到登录页 | SkillDetailView.js |
| B2 | 🟡 中 | Modal 组件 `show` prop 是死代码 (所有调用方都用 `v-if`)，且内置 cancel 按钮与调用方的 actions 重复，导致双重操作栏 | Modal.js |
| B3 | 🟡 中 | DocEditor 代码块格式化插入 `\\n` 字面量而非真实换行 | DocEditor.js |
| B4 | 🟡 中 | 图表用深色主题色 (`#ccc` 文字, 暗色网格) 渲染在浅色管理后台中，对比度差 | NewsAdmin.js |

### 9.2 设计不合理

| # | 严重度 | 问题 | 建议 |
|---|--------|------|------|
| D1 | 🟡 中 | SkillsView 全量加载所有 Skills 做客户端过滤，数据量大时性能差 | 改为服务端分页 |
| D2 | 🟡 中 | FileTree 模板硬编码只支持 3 层嵌套，更深目录不显示 | 改为递归组件 |
| D3 | 🟡 中 | DocEditor 作者硬编码为 "管理员"，未使用 `window.__currentUser` | 读取当前登录用户 |
| D4 | 🟢 低 | AppHeader 导航文字 "订阅管理" 指向公开 Feed 页，与"管理"含义不符 | 改为 "订阅动态" |
| D5 | 🟢 低 | `window.__currentUser` 全局状态脆弱，刷新后依赖重新请求 | 可接受，已有 /api/auth/me 恢复机制 |
| D6 | 🟢 低 | Skills API 缺少公开的 `/api/skills/:id` 端点 | 新增公开详情API |
