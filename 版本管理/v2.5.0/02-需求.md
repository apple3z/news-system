## 二、功能性需求

### 2.1 版本展示需求

#### 【需求编号】REQ-001
**需求名称**: 版本列表唯一性

**需求描述**: 系统管理页面左侧只显示一个版本树形菜单，禁止出现重复区域。

**用户场景**: 
> 作为用户，我打开系统管理页面时，希望看到清晰的版本列表，而不是两个重复的区域造成困惑。

**验收标准**:
- [ ] 页面DOM中只有一个版本树容器
- [ ] 版本数据只从 `/版本管理/` 目录扫描获取
- [ ] 删除 `versions.json` 缓存机制

**优先级**: P0

---

#### 【需求编号】REQ-002
**需求名称**: 版本号正确排序

**需求描述**: 版本按语义化版本号倒序排列，最新版本显示在最上方。

**用户场景**:
> 作为开发者，我希望最新版本显示在第一位，方便快速访问当前工作版本。

**详细规则**:
```
排序前（字符串排序）: v2.4, v2.5, v2.6, v2.10
排序后（数值排序）:  v2.10, v2.6, v2.5, v2.4

解析规则: "v2.5.0" → [2, 5, 0]
比较规则: [2, 10, 0] > [2, 6, 0] > [2, 5, 0]
```

**验收标准**:
- [ ] v2.10 排在 v2.6 之前
- [ ] 支持 2位/3位版本号（如 v2.5, v2.5.0）
- [ ] 支持带v前缀和不带前缀

**优先级**: P0

### 2.2 文档操作需求

#### 【需求编号】REQ-003
**需求名称**: 版本文档可访问

**需求描述**: 点击版本目录下的任意文档，能够正确加载到编辑器中。

**用户场景**:
> 作为开发者，我需要查看和编辑历史版本的文档，点击后应该立即加载内容。

**当前问题**:
- 混合使用 `/doc/` 和 `/版本管理/` 路径
- 某些情况下路径解析错误返回404

**解决方案**:
- 统一使用 `/版本管理/` 作为版本文档根目录
- 废弃 `/doc/` 路径（保留旧版本兼容）

**验收标准**:
- [ ] 所有版本文档点击后3秒内加载完成
- [ ] 文档内容正确显示在编辑器
- [ ] 加载失败时显示友好错误提示

**优先级**: P0

---

#### 【需求编号】REQ-004
**需求名称**: 文档列表实时同步

**需求描述**: Web端显示的文档列表必须与实际磁盘文件保持一致，实时反映增删改操作。

**用户场景**:
> 作为开发者，我在磁盘上新增了一个文档，刷新页面后应该立即看到它出现在列表中。

**当前问题**:
- 使用了内存缓存，未及时刷新
- 新增/删除文件后列表不更新

**解决方案**:
- 每次请求重新扫描目录
- 不使用缓存（数据量小，性能影响可接受）

**验收标准**:
- [ ] 新增文件后刷新页面立即显示
- [ ] 删除文件后刷新页面立即消失
- [ ] 重命名文件后列表同步更新

**优先级**: P0

---

#### 【需求编号】REQ-005
**需求名称**: 保存实时持久化

**需求描述**: Web端编辑保存后，内容必须立即写入磁盘文件，确保数据不丢失。

**用户场景**:
> 作为用户，我编辑完文档点击保存，希望内容立即安全存储到磁盘，而不是暂存在内存中。

**当前问题**:
- 保存操作可能异步执行
- 保存后立即刷新仍显示旧内容

**解决方案**:
- 同步写入磁盘后再返回响应
- 写入完成后清除相关缓存

**验收标准**:
- [ ] 保存API响应前文件已写入磁盘
- [ ] 保存后立即刷新显示新内容
- [ ] 文件修改时间正确更新

**优先级**: P0

### 2.3 模块边界需求

#### 【需求编号】REQ-006
**需求名称**: 明确模块职责边界

**需求描述**: 清晰区分"版本管理"和"文档中心"两个模块的职责，消除功能重叠。

**用户场景**:

|------|------|----------|----------|
| 版本管理 | 历史版本归档、只读浏览 | `/版本管理/v{X.Y.Z}/` | 只读（特殊情况可编辑） |
| 文档中心 | 现役文档编辑、日常维护 | `/docs/` | 读写 |

**验收标准**:
- [ ] 版本管理只显示 `/版本管理/` 目录内容
- [ ] 文档中心只显示 `/docs/` 目录内容
- [ ] 两个模块无功能重叠
- [ ] 界面明确标识当前所在模块

**优先级**: P0

### 2.5 目录结构规范需求

**需求描述**: Web界面显示的模块名称必须与服务器上的物理目录名称完全一致，消除歧义。
